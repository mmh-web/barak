03

Memory Deep Dive

Why File-Based Memory Is Powerful

Most AI assistants have one type of memory: the conversation window. Once it gets long enough, the oldest messages fall off. Details are gone forever.

Barak's memory works differently. The file system is the memory — and it has no upper limit. Every fact gets written to markdown on disk. Those files don't expire, don't get compressed, and don't fall off any edge.

The insight: this turns memory into a search problem, not a capacity problem. Search is a solved problem getting more efficient every day. As the platform develops, the file system is converging toward something that functions like a limitless context window.

In theory: infinite memory with perfect recall. In practice: we're not there yet.

Where It Breaks Down Today

The write problem.

Memory is only as good as what gets written. If Barak doesn't save it to a file, future sessions won't know. Saying "save that" helps.

The staleness problem.

Files can become outdated. No automatic expiry on a markdown file — it looks just as valid whether written yesterday or six months ago.

The search-not-read problem.

Search depends on asking the right question. If a fact is filed under an unexpected heading, Barak might miss it.

The confabulation problem.

The biggest failure: Barak thinking he remembers something when he's reconstructing from compressed summaries. The fix: always check the file.

What's Improving

  • Better automatic writes — detecting important facts and persisting without being told
  • Smarter search — semantic recall that works like natural questions
  • Lossless context compression — summaries that preserve critical details
  • File system as context — treating the entire workspace as context window extension

The architecture is right. The execution is catching up.

  • Set by: Michael (admin-level)

How to Edit: Just Talk

No config files to learn. Just tell Barak what you want:

# Valid ways to shape identity:

"Be more concise in briefings. Bullet points only."
"Don't use emojis in serious reports."
"I prefer you flag uncertainties explicitly."
"My timezone changed — I'm in NYC now."

# The richer, the better:
# Communication style, decision-making, ambiguity handling,
# specific phrases to use or avoid...

Barak updates the file. Takes effect next session.

The Drift Problem

Identity files are loaded at session start. As conversations get longer, identity instructions get proportionally smaller. Barak can drift from his configured personality.

Session start~95%
Mid-session (~50k tokens)~60%
Long session (~150k tokens)~25%

If Barak starts feeling generic — it's drift. A fresh session resets it. Or just say: "You're drifting, re-read your SOUL."

Tips for Working With Barak

  • Be explicit and say "write it down."
    "My timezone is PT, I'm in LA. Save that." — locks it into persistent memory.
  • Call out mistakes directly.
    Barak can check logs, configs, and state — but only if he knows something's off.
  • Think of requests as specs, not wishes.
    "Send me news at 6:30 AM PT, max 5 bullets, real journalism only" — that's implementable.
  • Ask him to build things.
    Not just Q&A — recurring workflows, monitoring scripts, data pipelines.
  • Don't assume he remembered something from weeks ago.
    If it wasn't written to a file, it may have faded. When in doubt, restate it.
  • Don't let him get away with vague answers.
    If Barak says "it should be working" without proof — push back.
